# План разработки приложения по методологии исследования компаний
### Версия 3.0 | Дата: 17 марта 2026

---

## Общая концепция

Приложение автоматизирует полный цикл стратегического исследования компании по методологии research.md:
- Сбор данных из всех каналов продвижения (сайт, VK, Telegram, карты)
- Анализ конкурентов по тем же параметрам
- Реверс-инжиниринг воронки продаж
- Генерация персонализированных рекомендаций через LLM с привязкой к бизнес-целям
- Экспорт финального отчёта

**Принцип разработки:** модульная архитектура — каждый модуль независим, имеет единый контракт данных (`SourceResult`), обрабатывает ошибки gracefully и не блокирует остальные модули при сбое.

---

## Целевые пользователи (Personas)

Определить до начала разработки — от этого зависит UX, детальность объяснений, приоритет фич.

| Persona | Сценарий | Приоритет функций |
|---------|----------|------------------|
| **Маркетолог агентства** | 5–10 отчётов в месяц, быстрый старт, шаблоны | Скорость, batch-режим, DOCX |
| **Фаундер/стартапер** | 1–2 отчёта в год, нужны объяснения терминов | UX-подсказки, понятный UI |
| **Консультант** | 3–5 отчётов в месяц, редактирует перед сдачей | Гибкий DOCX, история версий |

**Для MVP:** ориентируемся на маркетолога агентства — он даст быстрый feedback и понимает терминологию.

## Единый контракт данных

**Все парсеры возвращают одну и ту же структуру.** Это критично для сравнения, Gap Analysis и генерации рекомендаций.

```python
from pydantic import BaseModel
from typing import Optional, Dict, Any, Literal, List
from datetime import datetime

class SourceResult(BaseModel):
    source_type: Literal['website', 'vk', 'instagram', 'telegram', 'maps', 'competitor']
    status: Literal['success', 'partial', 'failed', 'skipped']
    trust_level: Literal[1, 2, 3]          # Уровень доверия из research.md
    collected_at: datetime
    data: Dict[str, Any]                   # Структурированные данные модуля
    errors: list[str] = []                 # Список ошибок/предупреждений
    disclaimer: Optional[str] = None       # Автодисклеймер для отчёта


class VerifiedMetric(BaseModel):
    """Метрика с верификацией из двух источников."""
    value: Any
    source_primary: str                    # Например: "vk_api", "2gis_api", "site_html"
    source_secondary: Optional[str]        # Второй источник для кросс-проверки
    confidence: Literal['high', 'medium', 'low', 'unverified']
    discrepancy: Optional[str] = None      # Описание расхождения, если есть
    verified_at: datetime


class Finding(BaseModel):
    """Единичное наблюдение/проблема/возможность по каналу."""
    channel: str                           # 'website', 'vk', 'instagram', 'maps', …
    severity: Literal['critical', 'warning', 'opportunity', 'ok']
    fact: str                              # Факт из данных, без интерпретации
    metric: Optional[VerifiedMetric] = None
    recommendation: Optional[str] = None  # Конкретное действие
    kpi: Optional[str] = None             # Измеримый результат
    priority: Literal['high', 'medium', 'low']
    effort: Literal['quick_win', 'strategic', 'longterm']


class ProjectResearch(BaseModel):
    project_id: str
    company_name: str
    business_goals: Dict[str, Any]         # Раздел 0 методологии
    sources: list[SourceResult] = []
    findings: list[Finding] = []           # Наблюдения по всем каналам
    competitors: list['ProjectResearch'] = []
    status: Literal['draft', 'in_progress', 'completed']
    created_at: datetime
    updated_at: datetime
```

### Обработка расхождений между источниками

| Тип расхождения | Критичность | Действие |
| --------------- | ----------- | -------- |
| Юридическое лицо, адрес, город | **Критичное** | UI-блок: жёлтое предупреждение с запросом подтверждения у пользователя |
| Количество отзывов (сеть vs точка) | Некритичное | `discrepancy` в `VerifiedMetric`, пояснение в отчёте |
| Разные цены на сайте и в картах | Некритичное | `discrepancy` + `Finding(severity='warning')` |

**Правило для критичных расхождений (UI):**

```text
⚠️ Обнаружено расхождение
Юридическое лицо на сайте: "ИП Тютюнова Ирина Андреевна"
Юридическое лицо в форме: "ИП Вайсберг"

Какое значение использовать в отчёте?
[Использовать данные сайта] [Использовать данные формы] [Указать оба]
```

## Принципы обработки ошибок (для всех модулей)

```python
# Три класса ошибок:

# FatalError — остановить весь анализ:
#   нет интернета, невалидный API-ключ, ошибка БД

# RecoverableError — пропустить источник, продолжить:
#   сайт недоступен, VK группа закрыта, rate limit, timeout

# PartialDataError — данные неполные, использовать с дисклеймером:
#   VK вернул 5 из 20 постов, на картах нет отзывов

# Паттерн для каждого парсера:
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=30))
async def fetch_with_retry(url: str) -> str:
    async with httpx.AsyncClient(timeout=15) as client:
        response = await client.get(url)
        response.raise_for_status()
        return response.text

# Checkpoint pattern — сохранять результат каждого модуля немедленно:
# если анализ упал на модуле 4, результаты 1-3 не теряются
# пользователь может повторить только упавший модуль
```

**В UI при ошибках:**
- `RecoverableError` → жёлтый статус: *"⚠️ VK группа закрыта. Анализ продолжен без этого источника"*
- `FatalError` → красный статус: *"Проверьте API-ключ VK в настройках"*
- `PartialDataError` → серый статус: *"ℹ️ Данные частичные — отчёт будет с дисклеймером"*

## Юридическое соответствие (обязательно изучить до старта)

| Платформа | Риск | Митигация |
|-----------|------|-----------|
| Яндекс.Карты | ToS запрещает авто-доступ | Использовать только для чтения публичных данных, не mass-парсинг |
| 2ГИС | Скрейпинг запрещён | По возможности официальное API |
| VK | Нельзя использовать данные в конкурентных целях без лицензии | Только анализ для самой компании |
| Telegram | Запрещено массовое скачивание | Ограничение: не более 30 сообщений на канал |
| 152-ФЗ | Имена авторов отзывов = ПД | При парсинге отзывов: сохранять только текст + рейтинг, имена — не сохранять |

**Добавить в каждый отчёт:**
> "Данные собраны в соответствии с публичным доступом платформ на дату анализа. Персональные данные авторов отзывов не собираются и не хранятся. Распространение собранной информации регулируется условиями использования соответствующих платформ."

### ЭТАП 3.2 — Модуль генерации рекомендаций (Finding-based + LLM)
**Приоритет: КРИТИЧНЫЙ ⚡⚡⚡**

#### Описание

Двухшаговый процесс: сначала `findings.py` детерминированно извлекает `Finding` объекты из `SourceResult`, затем LLM полирует язык рекомендации. LLM **не придумывает факты** — только улучшает формулировку готового Finding.

#### Задачи

**Шаг 1 — `findings.py` (детерминированная логика, без LLM):**
- [ ] Для каждого `SourceResult` — набор правил → `list[Finding]`
- [ ] Пример правила: `if vk.posts_per_week < 1 → Finding(severity='warning', fact='1 пост за последние 2 недели', recommendation='Публиковать минимум 3 раза в неделю', kpi='ER ≥ 2%', priority='high', effort='quick_win')`
- [ ] Gap Analysis: сравнение `Finding` компании и конкурентов, автогенерация GAP-Finding
- [ ] Классификация effort: `quick_win` (≤2 нед), `strategic` (1–3 мес), `longterm` (3+ мес)

**Шаг 2 — LLM (только тон и связность):**
- [ ] Claude API (основной) / YandexGPT (fallback)
- [ ] На вход: один `Finding` + контекст компании + бизнес-цель
- [ ] На выход: абзац текста для раздела отчёта (не структура данных)
- [ ] Промпт запрещает: добавлять факты не из Finding, использовать слова "улучшить/оптимизировать" без конкретики

```python
RECOMMENDATION_PROMPT = """
Ты редактор маркетингового отчёта. Перепиши Finding в один абзац для раздела отчёта.

Компания: {company_name} | Отрасль: {industry} | Бизнес-цель: {business_goal}

Finding:
- Канал: {channel}
- Факт: {fact}
- Рекомендация: {recommendation}
- KPI: {kpi}
- Приоритет: {priority}

Требования:
- Используй только факты из Finding — не добавляй новые данные
- Тон: профессиональный, конкретный, без воды
- Длина: 3–5 предложений
- Не начинай с "Рекомендуем" или "Необходимо"
"""
```

**Шаг 3 — Человек утверждает:**
- [ ] UI показывает каждый Finding: факт + сгенерированный текст рекомендации
- [ ] Кнопки: [Принять] [Редактировать] [Отклонить]
- [ ] Только принятые Finding попадают в финальный отчёт
- [ ] Статус сохраняется в БД (checkpoint)

**Контроль качества:**
- [ ] Проверка: все `Finding.recommendation` привязаны к бизнес-целям из формы ввода
- [ ] Проверка: нет `Finding` без `metric` когда данные доступны
- [ ] Удаление дублирующихся Finding по `(channel, fact)`

**Стратегия по этапам воронки:**
- [ ] Группировка Finding по этапам воронки (TOFU → Удержание)
- [ ] Quick Wins выделены в отдельный раздел (effort='quick_win')
- [ ] Долгосрочные — в дорожную карту

**Результат этапа:**
> ✅ Раздел рекомендаций с KPI — каждая рекомендация верифицирована фактом. Согласование.
